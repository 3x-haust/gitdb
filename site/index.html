Trade-offs

What GitDB is good at — and where it falls short

GitDB keeps query execution local and uses GitHub as a durable, inspectable audit trail. That makes it a good fit for some workloads and a poor fit for others.

Strengths

  • Every mutation is logged and visible in GitHub history
  • Local query execution — no per-query network round-trip
  • First-party ORM ships with the package, no extra provider needed
  • Encrypted storage option with AES-GCM out of the box
  • Manifest-gated commits with rollback-aware mutation logs
  • Good fit for agents, demos, config tools, and content pipelines

Limitations

  • ORM save() is slow — DELETE + INSERT per row in its own transaction (~46 writes/s)
  • GitHub Contents API writes are not a viable hot path
  • Not suited for high-throughput OLTP or concurrent multi-writer workloads
  • Limited SQL subset — not a mature general-purpose OLTP engine yet
  • No secondary indexes yet; joins scan every visible row

First-party ORM

Repository API for new apps

Define entities, open a data source, and use typed repositories. Each save() is upsert-safe but runs a DELETE + INSERT transaction — use raw dataSource.query() for bulk inserts. 

const Person = defineEntity({
  tableName: "people",
  primaryKey: "id",
  columns: { id: "STRING", name: "STRING", team_id: "STRING" },
})

const dataSource = await createGitDbDataSource({
  entities: [Person],
  store,
  synchronize: true,
})

await dataSource.getRepository(Person).save({
  id: "p1",
  name: "Ada",
  team_id: "runtime",
})
